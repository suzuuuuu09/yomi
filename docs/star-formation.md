# 要件・設計メモ

## 星の配置ロジック

計算はすべて **API側（Cloudflare Workers / Hono）** で行い、フロントには計算済みの値を返す。

### 座標の決め方

ジャンル別クラスタリングを採用する。ジャンルごとに3D空間の「星雲の中心」を固定し、同ジャンルの星が同じ領域に集まることで、ジャンル単位の「星座クラスター」を自然に形成する。

- ジャンルごとに中心座標 `(cx, cy, cz)` を決定論的に割り当てる（ジャンル文字列のハッシュから導出）
- 新しい星は中心から `±1.5` のランダム散布で配置
- 既存の星との距離が `< 1.0` になる場合はリトライしてオーバーラップを防ぐ
- 本の登録時にAPIが座標を計算してDBに保存する

### 色の決め方

ジャンル文字列から色相（Hue）を決定論的に固定し、読書進捗（brightness）で明度を変化させる。

- `genre → H`：ジャンル文字列をハッシュして `0〜360` の色相を算出（同ジャンルは常に同系色）
- `S`（彩度）: 固定 `70%`
- `L`（明度）: `brightness` に応じて `35%（未読）〜 75%（読了）` で変化
- 出力は `hsl(H, 70%, L%)` 形式の文字列
- 本の登録・進捗更新時にAPIが再計算してDBに保存する

### 星座の線を引く条件

「読了を星座の確定として演出」「ジャンルのメタデータで知識の関連性を線で結ぶ」というコンセプトに基づき、以下の3条件をすべて満たすペアに線を引く。

```
1. 両方の status === "completed"
2. 同じ genre
3. 3D距離 < 4.0
```

- 本が読了になるたびにAPIが `constellationLines` を再計算してフロントに返す
- フロントは受け取った `[string, string][]` をそのまま描画するだけでよい
